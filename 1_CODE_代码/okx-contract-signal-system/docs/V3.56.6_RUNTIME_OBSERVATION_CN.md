## 本轮审查发现

- v3.56.5 已经避免旧 `startup_quality_gate.json.stale_symbols` 覆盖在线运行态，但 Dashboard stale-symbol guard 仍有一个边界风险：运行态在线时，如果 closed backfill 已完整、币种状态为 `passed`，但 latest closed bar 的 wall-clock age 超过硬编码阈值，仍可能误报 stale。
- Dashboard 的 runtime health 已经把 fresh complete closed backfill 作为 quiet WebSocket 期间的有效行情活性证据；stale-symbol guard 也必须与这个权威来源一致。
- Dashboard 的 `runtime_approved` 口径原先只看 runtime online 与 `latest_scan.push_allowed`，没有把 15m closed backfill 完整性作为必要条件。若 future manifest 通过但 closed backfill 缺失，面板可能给出过宽的批准状态。

## 本轮修复

- stale-symbol guard 在 runtime online 时优先使用 `closed_kline_backfill_status.json` 的逐币种状态。
- 逐币种 stale 判定改为基于 `status != passed`、`missing_closed_bars > 0`、`data_complete=false` 或 `error`，不再对权威 closed-backfill 行单独使用裸 wall-clock age。
- runtime offline/stale/error 时仍回退到启动质检 stale 列表，用于诊断历史数据问题。
- Dashboard runtime approval 新增 15m closed backfill 完整性要求；缺失或未完成时加入 `closed_backfill_incomplete` 到 `push_blocking_reasons`。
- 新增前端测试覆盖：
  - fresh closed backfill 行不会因 `age_minutes > 90` 被误报 stale；
  - failed closed backfill 行会进入 stale-symbol 列表；
  - closed backfill 不完整时不能 runtime approve。
