# OKX 合约信号系统 v3.56.7 Dashboard 健康判定闭环

日期：2026-06-18

## 审查来源

本轮针对重新上传的同名 v3.56.6 发布包进行独立内容审查。该包的 ZIP 与随附 SHA-256 匹配，但与此前已验收的 v3.56.6 并非同一内容树；差异集中在 Dashboard stale-symbol、运行批准聚合、文档和发布元数据。

## 发现的问题

1. runtime 离线、stale 或 error 时，Dashboard 只返回旧 `startup_quality_gate.json.stale_symbols`，当前 closed-backfill 或 runtime symbol 失败可能被空旧列表掩盖。
2. runtime 在线但没有新鲜、完整的 closed-backfill 权威状态时，`status=passed` 且时间缺失/非法的币种可能被判定为健康。
3. closed-backfill 不完整但 manifest 已批准时，`approved_manifest_valid` 被错误加入 `push_blocking_reasons`，把通过条件显示成阻断原因。
4. 新鲜 closed-backfill 的权威覆盖没有检查状态新鲜度，旧的 complete 文件可能压过当前 runtime symbol 的 stale 证据。
5. 发布包内 `egg-info/SOURCES.txt` 由旧构建元数据生成，未完整列出当前发布文件。
6. `RealtimeDataStore.append_candle()` 对每次顺序追加都执行完整排序和去重，3500 根缓存保留回归在验收环境中超过数分钟，说明实时高频更新存在不必要的 CPU 开销。
7. 上传包将 15m 回补完整性加入 Dashboard `push_allowed`，但 Python 实际通知许可仍由 approved manifest 决定，可能出现面板显示拦截而后台实际允许发送的口径不一致。

## 修复

- 新鲜且完整的 closed-backfill 仅对其实际覆盖的币种具有权威性；未覆盖的配置币种仍使用当前 runtime 行判定。
- closed-backfill 过期、不完整或缺失时，恢复严格 runtime 行检查：失败、错误、缺少有效年龄或超过 90 分钟均进入 stale 列表。
- 非在线状态将启动质检、当前 runtime 行和 closed-backfill 失败取去重并集。
- manifest 通过时不再作为阻断原因；只有 `latest_scan.push_allowed=false` 时才加入真实 manifest 阻断原因。
- 拆分实际推送权限与运行健康：`quality.push_allowed`/`runtime_mode` 对齐 Python runtime 的实际 manifest 权限，`quality.operational_ready` 与 `health_blocking_reasons` 单独表达 closed-backfill 和 stale-symbol 健康状态。
- manifest 原因解析与阻断原因生成拆成纯函数并增加反例测试。
- 保留 v3.56.6 运行观察文档和 stale-symbol 审查文档，重新生成 Python 分发源清单。
- 实时 K 线合并增加有序快速路径：新时间戳直接追加、同一时间戳按 closed 优先替换；只有乱序、重复或非法时间数据才执行完整排序去重。
- 新增 closed K 线不能被后续 open 更新降级的回归测试；3500 根缓存保留测试恢复到可控时间内完成。

## 安全边界

- 本轮不修改信号策略、评分、风险计算、approved manifest 语义校验、SQLite outbox 或飞书发送主链。
- runtime cache 仅调整内存 K 线合并算法，不改变文件路径、closed 优先规则、最大缓存长度、历史只读边界或落盘格式。
- Dashboard 的 `quality.push_allowed` 与 `runtime_mode` 只反映 Python runtime 的实际 approved-manifest 推送许可；数据完整性和 stale 证据通过独立健康字段展示，不再伪装成后台权限门。
- 没有合法 strict-research approved manifest 时，系统继续处于研究观察模式。
