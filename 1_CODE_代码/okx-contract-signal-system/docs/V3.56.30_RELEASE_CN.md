# OKX 合约信号系统 v3.56.30 发布说明

发布日期：2026-06-28

## 本次目标

修复5分钟面板辅助补线任务在缓存长时间停更后，因逐币串行回补数千根历史空档而长时间卡住、状态文件不再刷新的问题。

本版本只调整5分钟运行缓存与健康检查，不改变15分钟正式信号数据、1小时趋势判断、策略参数、A级门禁、飞书策略、账户、持仓或下单边界。

## 问题表现

现场检查发现：

- 15分钟主补线正常，21个币种可自动重试并补到最新闭合K线；
- 桌面启动走GUI路径，而原5分钟服务只挂接在CLI信号循环中，GUI后台任务列表根本没有启动5分钟服务；
- `closed_kline_backfill_status_5m.json` 自2026-06-18后没有更新；
- 5分钟运行缓存中，部分币停在2026-06-18，部分币停在2026-06-21；
- 每个币尾部缺约2106至2854根5分钟K线；
- 原逻辑会先合并OKX最新300根，再把旧缓存与最新窗口之间的大段空档识别为内部缺口，逐币执行最多3000根回补；
- 21个币串行执行时，首轮可能持续很久，且完整周期结束前不刷新状态文件。

## 修复方案

### 1. 把5分钟服务正式接入GUI后台任务

GUI现在同时启动：

- 15分钟核心闭合K线补线；
- 5分钟面板辅助补线；
- 通知队列。

5分钟任务失败只会记录为辅助模块降级和警告，不会错误停止仍然健康的15分钟核心信号链。

### 2. 5分钟运行缓存改为最新连续窗口

`ClosedCandleBackfillService` 新增 `replace_with_latest_window` 模式。

当该模式启用时：

- 使用OKX返回的最新已确认闭合K线；
- 当缓存过旧、长度或时间窗口不一致时，原子替换为最新连续窗口；
- 不再尝试把停机期间数千根空档逐币补回运行缓存；
- 每轮仍校验最新闭合时间、内部间隔、连续尾部和写入结果；
- 状态增加 `latest_window_rebuilt` 字段，明确记录该币本轮是否重建窗口。

该模式仅用于 `okx_5m_extended` 面板辅助运行缓存。多年5分钟历史数据仍位于只读历史目录，不会被删除、覆盖或改写。

5分钟服务还为每个币种配置最多3次快速请求尝试、每次间隔1秒，用于吸收OKX REST的短暂超时；不会把重试扩展到长期无限循环。

### 3. 15分钟主补线保持严格模式

15分钟信号补线没有启用窗口替换：

- 继续合并新闭合K线；
- 继续检测和修复内部历史缺口；
- 临时OKX网络失败后继续在下一闭合周期自动重试；
- 任何未补齐币种仍会阻断核心运行健康门。

### 4. 增加5分钟健康告警

`scripts/system_check.py runtime` 新增三项非核心阻断检查：

- `dashboard_5m_backfill_fresh`；
- `dashboard_5m_backfill_complete`；
- `dashboard_5m_symbol_coverage`。

5分钟任务停更、写入失败或币种覆盖不完整时会显示明确警告，但不会错误阻断仍然健康的15分钟核心信号链。

## 回归测试

新增测试覆盖：

- 旧5分钟缓存与最新数据相隔数天时，直接重建最新连续窗口；
- 不触发长空档历史回补；
- 重建后无内部缺口、最新闭合时间正确；
- 15分钟原有内部缺口修复测试继续通过；
- 5分钟状态过期时，运行健康检查明确返回非阻断警告；
- 单个币第一次REST请求失败、第二次成功时，同一轮即可补齐并记录`attempts=2`；
- GUI后台任务列表确实包含5分钟服务，且启用最新窗口和3次快速重试。
