# OKX 合约信号系统 v3.56.3 运行缓存与安全闭环验收

日期：2026-06-18

## 结论

用户提供的 v3.56.2 包含有效的 runtime-cache 稳定性改进，但不能直接作为最终版，因为它覆盖掉了上一条修复线中的多项安全闭环：调度器重新硬编码允许推送、批准清单只校验哈希不校验研究语义、daily learning 再次写入正式候选文件、调度器不消费 outbox、15m 完整数据可能因非必要写失败被标成 failed，Dashboard 也重新混用旧快照。

v3.56.3 将两条修复线合并：保留 v3.56.2 的缓存稳定性，同时恢复并强化正式运行门、通知闭环和 Dashboard 可观测性。

## 主要修复

1. Runtime cache
   - 相同时间戳优先保留 confirmed closed K线。
   - 允许缓存保留一个当前实时 open 尾部。
   - REST closed K线可覆盖同时间戳的 open 行。
   - 新回补数据自身仍必须全部闭合。

2. 15m closed backfill
   - `data_complete` 与缓存写入结果分离。
   - 数据完整但非必要刷新写入失败时，仍保持 `status=passed` 和 `all_complete=true`。
   - 独立记录 `write_attempted`、`write_succeeded`、`write_error`、`write_failures`。

3. 正式批准清单
   - 同时校验哈希和研究语义。
   - 强制正式研究模式、v3.56-strict、blind sealed pass 且 `passed=true`、完整网格覆盖、可晋级、操作者、运行身份和审批时间顺序。
   - 仅重新计算哈希不能绕过正式门。

4. 参数与推送门
   - 调度器不再硬编码 `quality_gate_allows_push=True`。
   - 调度器和单次作业显式参数必须与批准清单完全一致。
   - 清单缺失、无效或参数不一致时保持 fail-closed。

5. Daily learning 隔离
   - 只写实验侧车 `daily_learning_candidate.json`。
   - 不再覆盖严格研究目录中的 `candidate_params.json`。
   - 正式运行不加载 daily-learning 候选。

6. Outbox
   - GUI、实时监控器和调度器均消费 SQLite outbox。
   - 调度器扫描后立即 drain，并在两次扫描等待期间继续 drain。
   - 失败重试达到阈值进入 DEAD_LETTER。

7. Dashboard
   - 最新扫描状态和 closed-backfill 状态为权威来源。
   - 旧三年回补报告只作最后兜底，不能覆盖最新 passed 状态。
   - 参数只显示运行时清单快照，不允许旧 quality/selected_params 覆盖。
   - 展示研究观察/正式推送模式、manifest 原因、扫描与回补年龄、outbox pending/failed/dead-letter/sent。
   - API 明确 UTF-8、no-store；时间按北京时间显示。

8. 发布与依赖
   - 恢复 PostCSS 8.5.10 override。
   - 发布包排除 `.next`、`node_modules`、TypeScript 缓存、outputs、数据库、环境密钥和 Python 缓存。

- TypeScript：通过。

- 结构反例：未发现硬编码放行、正式入口加载训练/回测/ML 决策、daily-learning 覆盖正式候选或 outbox worker 外直接发送。
